# /prep-pr

변경된 파일들을 PR 올리기 전에 점검한다. **커밋·푸시·PR 생성은 하지 않는다.** 점검 결과만 보고한다.

## 사전 준비

1. 프로젝트 루트에서 `git status`와 `git diff`(스테이징·언스테이징 모두)를 실행해 변경 범위를 파악한다.
2. base 브랜치(`main` 또는 `master`) 대비 변경이 있으면 `git diff main...HEAD`(또는 `master`)도 확인한다.

## 워크플로

### 1. 테스트 실행

- `npm test`를 실행한다.
- 결과를 아래 형식으로 요약한다:
  - **통과**: 실행한 테스트 수, 소요 시간(가능하면), 핵심 통과 내용
  - **실패**: 실패한 테스트 이름, 에러 메시지, 가능하면 원인 추정
  - **스크립트 없음/오류**: `package.json`의 `test` 스크립트 상태와 발생한 문제

**테스트가 실패하면** 2~4단계는 건너뛰고, 실패 원인과 수정 방향만 제안한다.

### 2. 변경 사항 요약 (테스트 통과 시)

- `git status`·`git diff` 결과를 바탕으로 **한국어 3줄**로 요약한다.
- 각 줄은 다음 관점 중 하나를 담는다: **무엇을** / **왜** / **영향·범위**
- node_modules, 빌드 산출물 등 PR에 포함되지 않을 파일은 제외한다.

### 3. 커밋 메시지 제안 (테스트 통과 시)

Conventional Commits 형식으로 **제목 1줄 + 본문**을 제안한다.

- 허용 접두사: `feat:`, `fix:`, `chore:`, `refactor:` 중 하나
- 제목: 50자 이내, 명령형, 한국어
- 본문: 변경 이유와 핵심 내용 2~4문장, 한국어
- 선택한 타입과 이유를 한 줄로 설명한다.

예시:

```
feat: 이메일 유효성 검사 및 목록 파싱 유틸 추가

사용자 배열에서 유효한 이메일만 추출하는 getValidEmails를 구현했다.
WHATWG HTML Standard 기준 정규식과 RFC 5321 길이 제한을 적용했다.
parseEmailList로 쉼표 구분 입력을 처리할 수 있게 했다.
```

### 4. 리뷰 포인트 (테스트 통과 시)

리뷰어가 **특히 확인해야 할 부분**을 2~4개 bullet로 짚는다. 아래 관점을 우선 검토한다:

- 핵심 로직·알고리즘 변경
- 엣지 케이스·입력 검증
- 테스트 커버리지 공백
- API·인터페이스 변경
- 성능·보안·호환성 trade-off
- 프로젝트 코딩 규칙(`.cursor/rules/`)과의 정합성

## 출력 형식

아래 템플릿을 따른다:

```markdown
## 테스트 결과

(요약)

## 변경 사항 요약

1. ...
2. ...
3. ...

## 제안 커밋 메시지

\`\`\`
(제목)

(본문)
\`\`\`

**타입 선택 이유**: ...

## 리뷰 포인트

- ...
- ...
```
